# SignalHive

## 社交交易智能平台

> 商业提案 | 2026年4月

---

## 痛点

加密货币交易者同时关注 Telegram 群、Twitter/X 大V、微博 KOL、YouTube 分析师等数十个信号源。现状：

- 同时手动监控 5-15 个渠道
- 人工判断哪些信号可执行、哪些是噪音
- 手动复制交易参数到交易所下单
- 无法追踪哪些博主真正盈利（大多数人不追踪）

**结果：** 错过信号、执行延迟、盲目信任无人问责的博主。

目前没有任何平台能让交易者 **自选信号源 → 自动提取可执行情报 → 验证 → 执行** — 一站式完成。

---

## 解决方案

**SignalHive** 是一个社交交易智能平台，将分散的社交媒体内容转化为结构化、评分化、可执行的交易信号。

```
接入渠道 → AI提取信号 → 评分排序 → 策略匹配 → 模拟交易 → 实盘上线
```

### 运作方式

1. **接入** — 用户添加自己已关注的 Telegram 群、Twitter 账号等渠道
2. **提取** — AI 读取每条消息，提取交易信号（币种、方向、入场/出场位、置信度）
3. **评分** — 多维评分：AI置信度 × 博主历史准确率 × 跨源共识度 × 时效衰减
4. **执行** — 信号自动匹配用户策略模板 → 先模拟交易 → 一键切换实盘

### 核心差异化

**博主准确率排行榜** — 我们追踪每个信号源的每条信号并衡量实际结果。交易者第一次能看到 *谁真正赚钱* 和 *谁在亏钱*。这些数据随时间积累，成为核心护城河。

## 技术

### 架构概览

```
┌────────────┐    ┌─────────────┐    ┌────────────┐    ┌───────────┐    ┌──────────┐
│  数据采集器  │ →  │  AI提取引擎   │ →  │   信号评估   │ →  │  策略匹配   │ →  │  交易执行  │
│ (按平台插件) │    │  (两层过滤)   │    │   (评分)    │    │  (模板)    │    │ (交易所)  │
└────────────┘    └─────────────┘    └────────────┘    └───────────┘    └──────────┘
```

- **插件化采集器** — 每个平台独立模块。从 Telegram 起步（已有基础），扩展到 Twitter、微博
- **两层提取** — 关键词预筛（免费）→ LLM 深度分析（按需调用）
- **多因子评分** — AI置信度 × 博主历史 × 跨源共识 × 时效衰减
- **Redis Stream** — 采集器与提取引擎之间的消息队列，保证可靠性

### 技术栈

| 层级 | 技术 |
|------|------|
| 后端 | Python / Flask |
| 数据库 | MySQL |
| 消息队列 | Redis Stream |
| AI/LLM | Claude API / OpenAI |
| 交易所 | ccxt（多交易所） |
| 前端 | Jinja2 (MVP) → React |
| 部署 | Gunicorn + Nginx + Supervisord |

### 现有基础设施（先发优势）

我们不是从零开始。关键组件已构建并在生产环境运行：

| 现有系统 | 复用 |
|---------|------|
| Telegram 信号追踪器 | → Telegram 采集器（已上线） |
| Trading SaaS 平台 | → Web 框架、认证、用户管理 |
| 交易机器人引擎 | → 模拟交易 & 实盘执行逻辑 |
| MySQL + ccxt 集成 | → 数据存储 + 交易所对接 |

**预计节省 4-6 周开发时间。**

## 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 假信号 | 错误交易，用户亏损 | 两层过滤 + 置信度阈值 + 人工确认模式 |
| 信号延迟 | 执行前价格已变 | TTL 机制，过期信号自动丢弃 |
| 博主失灵 | 系统继续信任失效信号源 | 滚动准确率 + 指数衰减 + 骤降告警 |
| 平台 API 变更 | 数据源中断 | 健康监控（5分钟心跳）+ 优雅降级 |
| 合规 | 法律责任 | Day 1 免责声明："不构成投资建议" + 风险披露 |
